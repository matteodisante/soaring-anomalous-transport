\section{The IGC tracklog format}

A flight track is stored as an IGC file, the standard plain-text format of the
F\'ed\'eration A\'eronautique Internationale for flight recorders. Each line is a typed
record whose \emph{type is fixed by its first letter}, a code defined by the standard rather
than chosen by the logger. The relevant codes are:
\begin{description}
  \item[\texttt{A}] the logger/manufacturer identifier (first record);
  \item[\texttt{H}] header records (date of flight, pilot, glider, datum, \dots);
  \item[\texttt{I}] the declaration of optional extra fields appended to each fix;
  \item[\texttt{B}] the GPS fixes -- the core of the file; ``\texttt{B}'' is simply the code
        the standard reserves for fix records.
\end{description}
A \texttt{B} record packs its fields by fixed character position. For example,
\begin{center}
\texttt{B\,094329\,4933044N\,00005193E\,A\,00095\,00149}
\end{center}
reads as: UTC time \texttt{09:43:29}; latitude $49^\circ33.044'$N; longitude
$000^\circ05.193'$E; a fix-validity flag (\texttt{A} for a valid 3-D fix, \texttt{V} for a
2-D one); and then \emph{two} altitudes, the barometric (pressure) altitude
(\SI{95}{\metre} here) and the GNSS/GPS altitude (\SI{149}{\metre}). The two differ in both
reference and noise -- the barometric value is referenced to a standard-atmosphere pressure
setting and is smooth, the GPS value is absolute (relative to the WGS84 geoid) but
vertically noisier -- and on older loggers one of them may be missing (recorded as zero).
Which altitude to use, and how, is left open and will be decided at the analysis stage
(Chapter~\ref{ch:next-steps}), because the vertical velocity derived from it drives the
phase segmentation.

The sequence of \texttt{B} records is therefore a time-stamped 3-D trajectory. The sampling
period is typically a few seconds but is not guaranteed to be uniform and varies with the
recording device -- a point the analysis stage must handle explicitly.

\section{The catalog}

Two derived tables make the raw files usable. The \textbf{catalog}
(\texttt{catalog.csv}) has one row per flight: the competition metadata plus a
\texttt{local\_path} column linking the flight to its file on disk, so a flight can be
traced to its track (and back, via the flight identifier embedded in the file name)
without any external lookup. The \textbf{season index} (\texttt{seasons\_index.csv}),
versioned in the repository, summarizes each season with the number of flights declared,
the number carrying a GPS track, and the number actually downloaded.

\section{Data quality and caveats}

A few limitations are already known and will shape the analysis:
\begin{itemize}
  \item \textbf{Flights without a track.} About \SI{8}{\percent} of entries have no IGC
        file (older or manually declared flights); they carry metadata only.
  \item \textbf{A small residue of unrecoverable files.} A few flights advertise a track
        that the server does not actually serve as a valid IGC (broken or placeholder
        links); these are recorded as failures and excluded.
  \item \textbf{Heterogeneous logging.} Sampling rate, altitude source, and accuracy
        depend on the device; trajectories must be resampled and quality-filtered before
        any ensemble statistic is computed.
  \item \textbf{Incomplete dates.} Some historical entries have placeholder dates (e.g.\
        \texttt{0000-00-00}); these are kept but flagged.
\end{itemize}
These caveats motivate the pre-processing described in Chapter~\ref{ch:next-steps}.
